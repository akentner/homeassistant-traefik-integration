<objective>
Add the two TLS entity platforms (`TraefikCertTimestampSensor` and
`TraefikCertExpiryBinarySensor`) under a new "HTTP Routers TLS" HA device,
plus a stale-cleanup listener that removes entities whose hostnames no
longer appear in any router's `tls.domains[]` + SAN + `Host()` set.

Purpose: This plan delivers TLS-01 (timestamp sensor with `notAfter` +
`days_until_expiry`) and TLS-02 (binary_sensor that flips ON when
`days_until_expiry <= threshold`, with live threshold re-evaluation).
The entities read from the `CertCoordinator` cache that plan 03-01 wired
into `entry.runtime_data.cert_coordinator` — no direct API calls.

Output: For each TLS-enabled hostname the user gets a sensor +
binary_sensor pair on the new "HTTP Routers TLS" device, ready for
dashboards, automations, and threshold-flipping UX.
</objective>

**Threat: Stale-state on transient cert cycle failure**
- The `_async_update_data` of `CertCoordinator` may fail (e.g., every
  probed host returns `CertError`). In that case
  `cert_coordinator.last_update_success = False` AND the cache may be
  stale.
- **Mitigation:** The cleanup portions of `_on_cert_update` in BOTH
  `sensor.py` AND `binary_sensor.py` gate on
  `cert_coordinator.last_update_success` before mass-deleting entities.
  Phase 2 D-18 pattern replicated verbatim. The timestamp sensor's
  `available` (delegating to `_cert_cache_availability`) returns False in
  this case — it doesn't crash, just shows "unavailable".

**Threat: Threshold-driven false negatives on entity state**
- A user lowers `tls_warn_days` from 14 to 7 expecting immediate impact.
  If the cert coordinator doesn't fire `async_update_listeners()`,
  entities stay in their last state.
- **Mitigation:** The `_async_options_updated` listener (plan 03-01 Task
  3) calls `cert_coordinator.async_set_threshold(new_threshold)` which
  in turn calls `async_update_listeners()`. HA's coordinator machinery
  re-renders every registered listener within the same event-loop tick.
  No data refetch — just state recompute.

**Threat: Privilege escalation via `_attr_entity_registry_enabled_default`**
- D-03 inverts the Phase 2 M-12 precedent by setting
  `_attr_entity_registry_enabled_default = True` for
  `TraefikCertExpiryBinarySensor`. This means users get the entity by
  default without opt-in.
- **Mitigation:** The entity is on a NEW device (`HTTP Routers TLS`) —
  even if a user has entity-registry auto-enable disabled at the
  HA-system level, no other devices' entities are affected. D-03 is
  documented in the class docstring; the user's CONTEXT.md decision is
  explicit ("auto-enable when threshold breached").
- If a user wants to opt out, they can disable the entity in HA's
  Settings → Devices & Services → Traefik → HTTP Routers TLS → the
  specific entity. Per-entity disable works regardless of the
  `entity_registry_enabled_default` value.

**Threat: Cleanup race during entity unregistration**
- A stale-cleanup listener might try to delete an entity while it's
  still being registered (race between `async_setup_entry` and the
  first cleanup tick).
- **Mitigation:** The `registry.async_remove(entity_id)` call is a no-op
  for unknown entity IDs (HA's entity registry handles this
  gracefully). No assertion or exception to catch.

**Threat: Hostname case sensitivity in unique_ids**
- Two routers could theoretically point at the same hostname with
  different casing (e.g., `Example.com` vs `example.com`). Phase 3
  normalises to lowercase inside `_collect_hosts_from_main_coordinator`
  — but if the cache is populated from a custom source (tests), a
  case mismatch could create two entities for the same host.
- **Mitigation:** Sensor and binary_sensor constructors defensive-lowercase
  the host (`host = host.lower()` at the top of `__init__`). This makes
  the unique_id idempotent regardless of the cache value's casing.

**Threat: `days_until_expiry` semantics confusion at hour boundaries**
- A cert with `not_after = 2026-07-06 23:59:59 UTC` shows
  `days_until_expiry = 0` until 23:59:59 the NEXT day (when it flips
  to `-1`). At hour boundaries, users see "0 days remaining" even
  though the cert expires within the hour.
- **Mitigation:** CONTEXT.md `<specifics>` documents this as a
  deliberate UX choice — floor at 0 for "still alive" cases to avoid
  alarming hour-boundary rounding. The timestamp sensor's `native_value`
  is the authoritative source for the exact expiry time; HA's TIMESTAMP
  device class renders both the date and time component.

**Threat: cert_coordinator missing (degraded install)**
- During a partial install or when the user patches a single file (e.g.,
  the integration is reloaded with a different config_entry mock in
  tests), `entry.runtime_data.cert_coordinator` may be `None`.
- **Mitigation:** Both sensor.py and binary_sensor.py `async_setup_entry`
  use `getattr(entry.runtime_data, "cert_coordinator", None)` and skip
  cleanly when None. No crash, no entities registered for the TLS
  side. The main integration still works.
